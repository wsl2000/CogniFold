## Executive Summary

As written, this is **not a 91–93 plan; it is a 89.5–91-ish plan with a
nontrivial chance of another hidden MS regression**. The core mistake is
that the plan still responds to failures by adding more global reader
rules, even though iter29/30 already showed that ungated rule accretion
is your most reliable way to destroy MS.

The highest-ROI missing move is **not another rule**: it is a **gated
count/order/date-diff answer path that builds a small evidence table
before answering**. Without that, `EXHAUSTIVE-COUNT v2` is mostly
wishful thinking, `rerank 100→200` is a tax rather than a fix, and
beating Mastra's published headline in one more cycle is very
unlikely. Also, two of the remaining TR misses are publicly disputed
benchmark-quality cases, so the true engineering ceiling is lower than
the raw wrong-count suggests.

---

## 1. Highest-ROI fix you are missing

A **question-shape router with a structured evidence pass**, implemented
in `run_eval.py`, not more global text in `longmemeval_profile.yaml`.

Detect: `how many`, `total hours`, `what order`, `since/before/after`,
and `_abs` questions. Force a separate path that first emits a compact
JSON/evidence ledger like `[{item, date, value, evidence_id}]`, then
counts / orders / refuses.

Why best ROI: your dominant MS cluster is not "reader lacks one more
hint," it is "reader stops after the first 2–3 hits." A ledger step
directly attacks that. It also helps TR order cases because the model
stops answering from prose gestalt and starts answering from an
explicit candidate list.

The real lesson from iter29/30: the problem was not specialization,
it was **ungated specialization inside one shared prompt**.

## 2. Most regression-prone proposed fix

**`EXHAUSTIVE-COUNT v2` as a global qa_answer rule.** Mechanism: it
will push the reader to over-list semantically related items, count
from partial matches, and fight abstention/refusal cases. Exactly the
"sounds good in isolation, contaminates everything else" rule that
already burned you in iter29/30.

Second-highest risk: **rerank pool 100→200 for "how many"**. Increases
distractors without changing retrieval granularity. More noisy concepts
in context often worsen counting (more near-matches to accidentally
include, more chances to lose exact boundaries).

## 3. PAL / code-augmented counting — yes, narrow MVP only

2–3 hour MVP:

```
count/order question
 -> retrieve context
 -> build evidence rows {normalized_item, date, numeric_value, source_ref}
 -> Python dedupe / tally / sort
 -> hand the computed result + evidence rows to the reader for final wording
```

Gated, auditable, directly hits the 22-case MS-A cluster. Do NOT build
a generic PAL agent. Build a tiny deterministic tally helper. If you
cannot extract rows deterministically, let the model emit JSON first
and reject any row not grounded in retrieved context.

## 4. Mastra's edge

The useful Mastra trick is **not** triple-date annotation or priority
tagging in isolation. It is **writer-time distillation into a stable,
chronological observation log that the actor can mostly read
end-to-end, plus reflection/merging over that log**. That removes the
retrieval-unit problem and preserves temporal order before answer time.
You are still asking the reader to rediscover the answer from retrieved
graph fragments.

The gap: "their answerer consumes curated chronological memory; yours
consumes retrieved pieces and hopes the reader reconstructs chronology
correctly."

Also: Mastra's published category counts visibly sum to `468/500 =
93.6%`, not 94.87%. Treat `94.87` as a headline number, not a cleanly
audited denominator. Source: <https://mastra.ai/research/observational-memory>

## 5. Structural retrieval fix for both TR and MS

**Chunk-level late fusion over raw session text**, not "bigger pool."

Keep graph retrieval, but union it with a cheap lexical/BM25 retriever
over raw message/event chunks or per-event snippets, then rerank the
union. Trigger especially for count/order/time-anchor questions or
whenever graph hits do not produce enough unique candidates.

Helps the missing-`JetBlue`, missing-`Museum of History`, the correct
"last Saturday" event, and "two weeks ago" art event — and the MS
undercount cases where the missing item exists in the source session
but got merged away. Public LongMemEval retrieval artifacts strongly
point to **retrieval granularity** as the lever, not just more top-k:

- <https://github.com/xiaowu0162/LongMemEval/issues/46>
- <https://github.com/xiaowu0162/LongMemEval/issues/34>

## 6. Run order

If two full runs are affordable: **run N=500 on the current iter31
stack first**. The current plan is too projection-driven. "It should
track iter19 MS" is not evidence; `gpt-5.4-mini + new reader rules +
X1 + activity_start` is a different system. Get the real
MS/SSA/SSP/SSU/KU baseline before adding more rules.

If only one full N=500 is affordable: ship ONLY these two changes into
that run:
1. The gated count/order evidence path
2. The chunk-level late-fusion retrieval fallback

Everything else is lower expected value or too regression-prone
relative to the information you still lack.

## 7. `_try_date_diff_since` calendar-month patch

Smallest safe patch — change ONLY the `month/months` branch in
`symbolic_resolver.py`, leave day/week behavior untouched:

```python
if unit in {"month", "months"}:
    months = (end.year - start.year) * 12 + (end.month - start.month)
    if end.day < start.day:
        months -= 1
    return months
```

Why smallest: fixes `b46e15ed` without perturbing branches that are
already passing. Do NOT generalize into weeks or generic "duration"
rounding. The bug is specifically that `days/30` is the wrong
abstraction for month questions.

## 8. `_find_is_start_concept` Pass 3

The problem is **candidate selection**, not rounding. Inclusive-boundary
convention can explain `±1`; it cannot explain using the wrong anchor
concept.

An EARLIEST-mention fallback is semantically wrong for
recovery/state-transition questions: "first time flu was mentioned"
≠ "when I recovered from the flu."

Right fix: **limit Pass 3 fallback to genuine start/activity-start
cases**. Do NOT use it for recovery/end-state/event-anchor questions.
For "recovered / got over / healed from," require an explicit recovery
concept/date or fall back to reader reasoning over raw context. If you
patch this by changing week rounding to `ceil`, you are treating the
smoke, not the fire.

Also: `370a8ff4` is **publicly disputed** as a benchmark labeling
problem (<https://github.com/xiaowu0162/LongMemEval/issues/41>).
`eac54add` has a public annotation-quality dispute
(<https://github.com/xiaowu0162/LongMemEval/issues/37>). Do not
contort core logic just to force these specific answer shapes.

---

## Bottom Line

The round-2 plan is **too broad, too prompt-heavy, and too optimistic**.
Expected-value version:

1. Baseline first if you can afford it.
2. Then make **one gated answer-path change** (question-shape router +
   structured evidence ledger).
3. **One retrieval-unit change** (chunk-level late fusion over raw
   session text).

If you keep piling global rules into the shared reader prompt, you are
repeating the exact failure mode that already cost you a full iteration.
